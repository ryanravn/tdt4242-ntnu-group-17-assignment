# Task 3.4 - Responsible AI Use Reflection

## Description

In this task, you will produce a structured reflection on your use of AI tools throughout the entire software development lifecycle of the project. The reflection should demonstrate critical thinking, technical understanding and awareness of how AI influenced both process and professional judgement.

Your reflection must address the following phases:

1. **Requirements Elicitation with AI vs. Human Stakeholders** - Reflect on your experience using the iRE Agent compared to interviewing a real human stakeholder. Discuss differences in depth, clarity, assumptions, follow-up questioning and limitations of each approach.

2. **Using AI to Understand Web Development Concepts** - Explain how AI tools supported (or failed to support) your understanding of web development concepts, architectures, or frameworks. Include examples where AI helped clarify concepts or where independent validation was required.

3. **AI-Assisted Code Generation Process** - Describe how you used AI tools to generate code, including prompting strategies, iteration patterns and how requirements were translated into prompts.

4. **Revision and Correction of AI-Generated Code** - Analyse how and why you modified AI-generated code. Discuss typical issues encountered (e.g. logic errors, missing edge cases, architectural weaknesses) and explain your reasoning when fixing or refactoring the code.

5. **Quality Assessment of AI-Generated Code** - Evaluate the overall quality of AI-generated code in terms of correctness, readability, structure, maintainability and alignment with requirements.

6. **Best Practices Developed When Working with AI** - Identify the working practices you found most effective when using AI in this project, such as trial-and-error learning, analogy-based prompting, guided reasoning, stepwise refinement, or rapid feedback loops. Explain why these practices worked.

7. **Professional Identity and Way of Working** - Reflect on how working with AI affected your role as an engineer. Did AI increase or reduce your sense of responsibility, confidence, or judgement? Discuss whether AI supported deeper engineering thinking or encouraged superficial solutions.

## Reflection

### 1. Requirements Elicitation with AI vs. Human Stakeholders

The iRE Agent was highly useful in the early elicitation phase because it provided fast, structured and policy-oriented answers. This made it easy to convert its responses into clear functional and non-functional requirements. However, the interaction also felt "too clean", since the AI rarely showed the ambiguity, hesitation, or conflicting priorities often present in real human communications. The human stakeholder interview was less structured and required more follow-up questions, but it revealed more realistic concerns, such as confusion around institutional AI rules, fear of accidental misuse and privacy expectations. The strongest requirements ultimately came from combining AI's efficiency and consistency with the realism and depth of human stakeholders.

### 2. Using AI to Understand Web Development Concepts

AI tools were very useful for quickly understanding full-stack web development concepts such as frontend-backend separation, API request flow, routing and component structure. Because not every member of the group had prior web development experience, AI significantly reduced the learning curve and helped establish an early understanding of the system architecture. It helped build a clearer mental model of API logic, backend endpoints and database interactions, especially when learning unfamiliar frameworks and local testing workflows. It is worth mentioning that some of the generated answers were too generic, sounding more convincing than they were actually applicable to the project. Code and architectural suggestions therefore had to be deleted or modified based on selected requirements and assignment constraints.

### 3. AI-Assisted Code Generation Process

AI was used extensively for generating test infrastructure and documentation. The prompting strategy was task-driven: the task description from the course assignment was provided to the AI, which then generated the corresponding deliverable. For test planning (Task 3.1), AI was given the requirement IDs (RE-09 through RE-16) and produced a full test case table, traceability matrix and coverage definitions. For test execution, AI generated manual test scripts, added TC-ID comment headers across all test files and created the complete E2E Playwright suite from the test case IDs defined in the plan. The iteration pattern was consistent: AI generated an initial version, it was run locally, the terminal output was fed back and AI refined based on the actual errors. This feedback loop was repeated multiple times, particularly for the Playwright configuration.

### 4. Revision and Correction of AI-Generated Code

The most significant correction effort involved the Playwright E2E setup, which went through six or more revision cycles. AI initially generated a TypeScript config (`.ts`), which failed because Playwright's internal ESM preflight transform did not work under Bun's runtime. AI then switched to `.mjs`, which failed for the same ESM reason. A `.cjs` config was tried next, but a block comment containing `**/*.spec.ts` included `*/` which terminated the comment and broke parsing — an error AI introduced and then had to fix. The `testMatch` glob also failed to match files because Playwright's bundled minimatch did not handle the default extglob pattern, requiring an explicit simple glob. The root `"type": "module"` in `package.json` caused Playwright to load test files via the ESM path even with a CJS config; this was resolved by adding a separate `e2e/package.json` with `"type": "commonjs"`. The E2E test logic also needed three rounds of fixes: the DEV-mode auto-login interfered with the login flow, `getByText` matched two elements causing a strict mode violation and `localStorage` access failed on `about:blank`. These issues demonstrate that AI-generated code for complex tool integrations often requires substantial human debugging.

### 5. Quality Assessment of AI-Generated Code

The quality of AI-generated output varied significantly by task type. Documentation and test planning were high quality on the first pass — the test plan, traceability matrix and 3.2 report were comprehensive, well-structured and directly traceable to requirements. Manual test scripts also worked correctly without revision. However, the Playwright configuration was poor quality initially, requiring six iterations to function correctly under Bun's runtime. AI did not account for the interaction between Bun, ESM module resolution and Playwright's internal transform pipeline and repeatedly proposed fixes based on incomplete understanding of the failure. The E2E test code was structurally correct but made wrong assumptions about the application's DEV-mode auto-login behavior. The final test suite achieved 97.30% line coverage and 85.19% function coverage with all 38 tests passing, indicating that the underlying test logic was solid once the configuration issues were resolved.

### 6. Best Practices Developed When Working with AI

The most effective practices developed were staged prompting, explicit constraint setting and rapid validation after each generation step. Instead of asking AI to generate complete solutions immediately, it was first used to clarify requirements, then map them to architecture and only afterwards move into code generation and review. When team members encountered concepts they did not fully understand, AI was deliberately asked to explain them in simple, step-by-step language before continuing, which helped reduce misunderstanding in the process. It was also found valuable to treat AI not only as a generator but also as a reviewer, asking it to justify design choices, identify code smells and compare outputs against requirements. A critical practice that emerged was the feedback loop: running AI-generated code immediately, copying the exact terminal error back to AI and letting it iterate. This worked well for straightforward errors but was less effective for systemic issues like the Bun-Playwright compatibility problem, where AI needed multiple attempts because it lacked runtime context.

### 7. Professional Identity and Way of Working

Working with AI changed the team's role from mainly writing code to more actively defining constraints, validating outputs and making engineering decisions. While AI reduced the time spent on boilerplate implementation and helped the team move faster, it also increased the importance of human judgement. The team became more responsible for checking whether generated code truly matched requirements, handled edge cases and remained within the project scope. In this sense, AI did not reduce responsibility as engineers, but rather shifted it toward verification, architectural thinking and critical decision making. The Playwright debugging experience was particularly illustrative: AI generated plausible-looking fixes that failed repeatedly and it was ultimately the human understanding of the runtime environment (Bun vs Node, ESM vs CJS) that guided the solution. The most valuable effect was that AI allowed less time to be spent on repetitive implementation and more time on deeper reasoning about system quality and design choices.
