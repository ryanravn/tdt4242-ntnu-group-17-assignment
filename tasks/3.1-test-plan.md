# Task 3.1 - Test Plan

## Description

In this task, you will design a structured test suite for the AIGuidebook system that reflects professional quality-assurance (QA) practices. Your test set must include a balanced combination of unit tests, integration tests, system tests, and non-functional requirement (NFR) tests.

You will also define test coverage criteria for the system, including coverage of lines of code (LOC), branches/paths, and requirements. In addition, you must create a traceability matrix that links each requirement group selected from Task 2.1 to one or more test cases and, where possible, to specific implementation components.

## Expected Outcome

A test plan document including:

1. **Test Case Table** (cover all the given requirement group) using the following structure:

| Test ID | Test Type | Input / Trigger | Expected Output | Test Method | Pass/Fail Criteria | Notes |
|---------|-----------|-----------------|-----------------|-------------|-------------------|-------|

2. **Coverage Definition** briefly specifying coverage goals, for example:
   - LOC coverage (e.g. >= 65%)
   - Branch/path coverage (e.g. each conditional evaluated true and false)
   - Requirement coverage (100%)
   - NFR coverage (at least one test per NFR category, such as security or usability)

3. **Traceability Matrix** linking: Requirement -> Test Case(s) -> Implementation reference (file or component, if applicable)

## Evaluation Criteria

- **Breadth and depth of test coverage:** A well-balanced mix of unit, integration, system, and NFR tests that validates the full requirement set rather than isolated examples.
- **Technical correctness of test cases:** Each test clearly specifies inputs, actions, expected outcomes, and testing method or tool.
- **Professional use of traceability:** The traceability matrix clearly maps requirement IDs to concrete test cases, demonstrating complete requirement coverage.
- **Quality of coverage metrics:** Coverage goals are meaningful and justified (e.g. branch, line, requirement, and UI flow coverage), not merely listed.

## Coverage Definition

### Line of code (LOC) coverage

- **Goal:** ≥ **70%** statement/line coverage on `server/routes/**`, `server/lib/auth.ts`, and risk/classification logic (whether kept in `declarations.ts` or extracted).
- **Justification:** The business value is concentrated in the API layer and auth; 70% is achievable without testing generated boilerplate, while forcing coverage of happy paths and main error branches. Tooling: `bun test --coverage` when enabled, or Istanbul/c8 compatible with the Bun runner.
- **Exclusions (documented):** `server/db/seed.ts` (one-off script), static Scalar/OpenAPI wiring in `server/index.ts` (smoke-tested via TC-INFRA-02).

### Branch / path coverage

- **Goal:** For **risk classification** (`classifyDeclaration`), every branch that sets `riskLevel` is exercised at least once (high via direct_answers, high via undeclared tools, medium via substantial tasks or frequency, low otherwise). For **logs GET**, each filter combination: no query, `assignment_id` only, `from`/`to` only, combined (as applicable).
- **Justification:** Branch coverage catches “dead” conditions and wrong `else` ordering; the risk tree is the most fault-prone domain logic.

### Requirement coverage

- **Goal:** **100%** of RE-09–RE-16 mapped to at least one test case (see traceability matrix). Each acceptance bullet should be covered by at least one **Integration** row unless explicitly deferred to **System** (UI-only wording).

### NFR coverage

- **NFR-01 (logging responsiveness):** TC-NFR-01 (performance category).
- **NFR-02 (API response time):** TC-NFR-02 (performance category).
- **Security (implicit):** TC-AUTH-02–07, TC-RE09-05, TC-RE12-06, TC-NFR-SEC-01 — authentication and authorization expectations.
- **Usability:** TC-SYS-01–TC-SYS-04 — validates end-to-end flows that match “quick logging” and transparent declaration workflows described in user stories.
